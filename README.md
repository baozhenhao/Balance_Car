本项目是一款基于 STM32 微控制器的自平衡两轮小车。项目不仅实现了底层的传感器驱动与运动控制，更完整记录了全流程的软硬件联调与“破案式”故障排查过程。

## 📌 项目核心亮点
* **代码分层架构**：基于 C++ 面向对象思想封装硬件外设，驱动层、业务层、控制层完全解耦，拒绝面条代码。
* **工程版本控制**：严格遵循 Git 规范进行阶段性版本提交，具备清晰的代码演进历史。
* **硬核调试流**：具备独立的软硬件联合调测与物理故障排查能力。

---

## 🛠️ 硬件架构与选型
* **主控芯片**：STM32F103C8T6
* **姿态传感器**：MPU6050（六轴陀螺仪）
* **电机驱动**：TB6612FNG 双路电机驱动板
* **执行机构**：带增量式光电编码器的直流减速电机
* **通信模块**：蓝牙透传模块（用于数据回传与远程调参）

---

## 🚀 核心控制算法
* 采用经典 **PID 串级控制算法**（直立环 + 速度环）。
* 状态反馈周期：严格控制在 5ms 核心控制周期内，确保小车姿态响应的实时性与动态稳定性。

## 🔍 软硬件联调与“破案”日志 (Engineering Logs)

本项目的核心价值不仅在于最终能够实现平衡，更在于解决了一系列经典的嵌入式开发“暗坑”：

### 1. 环境配置坑：编译器被本地安全中心误杀
* **问题现象**：合入 SSD 驱动库后，IDE 频繁报 `Error 7` 严重错误，提示找不到编译器。
* **排查过程**：排查路径与代码逻辑无果后，发现由于第三方库触发了 Windows 安全中心的误报机制，导致工具链核心组件被直接隔离。
* **解决方案**：将整个 STM32CubeIDE 工作区（Workspace）加入安全中心信任区，并执行 Project Clean 重新编译，环境恢复正常。

### 2. 通信底层坑：硬件 I2C 锁死故障
* **问题现象**：MPU6050 传感器数据无法上报，且串口/蓝牙持续输出乱码。
* **排查逻辑**：由于 STM32 硬件 I2C 存在经典的偶发性锁死 Bug（总线被从机拉低），导致软件卡死在等待信号的死循环中。
* **解决方案**：放弃脆弱的硬件 I2C，重构为**软件模拟 I2C 驱动**。引入超时退出机制，彻底解决了总线锁死问题，蓝牙透传数据恢复正常。

### 3. 硬件链路坑：编码器信号缺失与单片机引脚损坏（经典案例）
* **问题现象**：软硬件联调期间，左轮编码器（Encoder）脉冲读数始终为 0，小车无法闭环。
* **控制变量排查过程**：
  1. **排查电机本身**：将左右两个轮子进行对调交叉测试，发现左侧数据依旧为 0，右侧正常 ➔ **排除电机及编码器本身损坏**。
  2. **排查驱动及接线**：更换全新的 TB6612 连接线，且通过代码控制 TB6612 发现能够正常操纵左轮电机转动 ➔ **排除驱动板与动力线缆问题**。
  3. **锁定主控异常**：信号链路一切正常，唯一可能的故障点在于 STM32 的对应定时器输入捕获 GPIO 引脚发生了不可逆的物理损坏。
* **解决方案**：修改底层配置，启用 **TIM3 的引脚复用（Remap）功能**，将输入捕获通道平移到全新的 GPIO 引脚。重新烧录后，成功读取到左轮编码器脉冲数据，完美解决故障。
